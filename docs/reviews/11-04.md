## 2. Lo que está bien hecho

### Documentación de arquitectura (`docs/architecture.md`)

- Esquema relacional bien pensado con 8 tablas, relaciones claras e índices específicos.
- La separación SQLite (dev) / PostgreSQL (prod) es pragmática para un MVP.
- El campo `entity_id` como enlace entre PostgreSQL y ChromaDB es una decisión acertada.

### Estrategia RAG (`docs/data-sources.md`)

- La especificación v3.0 es ambiciosa y bien detallada: hybrid search (0.7 vector + 0.3 BM25), cross-encoder re-ranking, entity resolution con niveles de confianza.
- El modelo canónico de datos con lineage (checksum SHA-256, `ingestion_id`, `data_version`) es excelente para trazabilidad.
- La detección de conflictos regulatorios (FDA vs EFSA) con matriz de severidad es el diferenciador clave del producto.

### PRD v5.0

- El semáforo visual (Gray/Blue/Yellow/Orange/Red) es una buena abstracción UX para comunicar riesgo.
- El flujo de failover (barcode → foto de etiqueta) muestra pensamiento de resiliencia.

### Convenciones (`.claude/CLAUDE.md`)

- Reglas claras: Pydantic v2, Structured Outputs de Gemini, JWT obligatorio, AES-256 para datos médicos.

## 3. Problemas y Riesgos Identificados

### 3.1 — No hay código implementado

Hallazgo principal. Después de 5 commits y documentación extensa, los 15 archivos de código fuente están vacíos. Hay riesgo de **parálisis por planificación**: se ha invertido todo el esfuerzo en especificación sin validar nada con código real.

### 3.2 — Archivos críticos de configuración vacíos

| Archivo                      | Impacto                                                      |
| ---------------------------- | ------------------------------------------------------------ |
| `requirements.txt`           | No se puede instalar nada                                    |
| `Dockerfile`                 | No se puede contenerizar                                     |
| `.env.example`               | No documenta qué variables son necesarias                    |
| `docs/prompts.md`            | Los prompts son el corazón del sistema agéntico y no existen |
| `docs/embedding-strategy.md` | Referenciado en la documentación pero vacío                  |

### 3.3 — Problemas de arquitectura

#### Base de datos

- La tabla `biomarkers` usa `BYTEA` para `encrypted_data` pero **no hay campo para el IV (Initialization Vector)** del AES-256. Sin el IV, no se puede desencriptar. Necesita un campo `encryption_iv`.
- `expires_at` con TTL de 180 días en biomarkers: no hay mecanismo definido para el cron job de limpieza (mencionado pero no especificado).
- **Falta tabla `products`** — `scan_history` guarda `product_barcode` como texto plano sin normalizar. Si un producto se escanea 1,000 veces, se duplica el barcode 1,000 veces sin datos del producto asociados.

#### Seguridad

- El PRD menciona JWT via HTTP-only cookies, pero CLAUDE.md solo dice "JWT". Falta definir refresh token strategy, token rotation y revocación.
- No hay mención de **rate limiting** en ningún endpoint — crítico para un servicio que procesa imágenes con un LLM externo (costo por llamada a Gemini).
- No se define cómo se genera/almacena la **clave AES-256**. ¿Variable de entorno? ¿KMS? Esto es fundamental para datos médicos.

#### RAG

- El template de embedding dice "Max 256 tokens" pero el chunking dice "512 tokens". Hay **inconsistencia** entre el tamaño del chunk y el embedding template.
- El umbral de 0.7 para HITL (Human-in-the-Loop) en entity resolution es arbitrario y no tiene plan de calibración.
- No hay estrategia de fallback si ChromaDB está caído.

### 3.4 — Falta el frontend completo

El stack dice "Next.js" pero no existe carpeta `frontend/` ni ningún archivo de frontend. Ni siquiera el scaffolding.

### 3.5 — No hay tests

No existe ninguna estructura de testing — ni `tests/`, ni `pytest.ini`, ni `conftest.py`. Para un sistema que maneja datos médicos, la ausencia total de estrategia de testing es preocupante.

### 3.6 — No hay CI/CD

No hay GitHub Actions ni pipeline de ningún tipo. Para un proyecto que menciona deployment en Render/Railway, debería al menos estar scaffoldeado.

### 3.7 — README.md vacío

El README es la puerta de entrada del proyecto y está vacío.

## 4. Problemas del PRD

- **Scope creep potencial**: El MVP intenta cubrir OCR de etiquetas + búsqueda semántica + análisis de biomarcadores + detección de conflictos regulatorios + encriptación médica. Eso es demasiado para un MVP.
- **Sin métricas de éxito**: No define qué constituye "funciona" — ¿accuracy del OCR? ¿precision/recall del RAG? ¿latencia objetivo?
- **Sin user stories**: Hay workflows técnicos pero no user stories que validen que el flujo tiene sentido para el usuario final.
